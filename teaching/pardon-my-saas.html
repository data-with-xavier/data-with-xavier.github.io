Before we write a single line of code, let's talk about why you're here — and why the tools your company probably uses are working against you.

The SaaS Tax

Tableau Desktop runs $840/year per user. Power BI Pro is $120/year but requires a Microsoft 365 ecosystem. Looker? Enterprise pricing — call for a quote (which means very expensive). Every year, companies pay these "SaaS taxes" for tools that lock your work into proprietary formats, limit your design choices, and slow you down the moment you want to do something outside their feature set.

The lock-in is the real cost. Your Tableau workbooks can't run without Tableau. Your Power BI dashboards live in Microsoft's cloud. You're renting access to your own work.

The AI Alternative

HTML is the language of the web. It's free, runs everywhere, and looks exactly how you design it. The catch has always been: you need to know how to write it. That's where Claude changes everything.

You describe what you want. Claude writes the code. You refine it by describing what to change. No syntax memorization. No Stack Overflow spirals. Just a conversation that produces something you can publish.

This isn't "AI does everything while you watch." You are the analyst, the designer, the storyteller. Claude is the developer who executes your vision faster than any team could.

It means you're done apologizing for not knowing Tableau. It means you're building something better, on your own terms, that you actually own. By the end of this course, you'll have a portfolio of live, published dashboards — not screenshots from someone else's platform.

How Context Works — and Why It Matters

Every Claude conversation starts completely fresh. Claude has no memory of previous sessions. When you start a new chat, it doesn't know what you built yesterday, what colors you prefer, or what your portfolio looks like.

This trips up a lot of new users. They open a new Claude Code session, type "add a filter to my dashboard," and wonder why Claude seems confused. It's not confused — it just has no idea what dashboard you're talking about.

Important: Starting a new Claude session mid-project without context is like calling a contractor and saying "finish what you started" — except they have no memory of ever meeting you. Always re-orient Claude at the start of each session.

What to do about it:

  • Start new Claude Code sessions by asking Claude to read your existing files first: "Read my index.html, then I'll tell you what to add."
  • The /dash skill you'll build in Module 04.5 exists specifically to solve this — you'll invoke /dash at the start of every new build session so Claude immediately knows your style and walks you through planning before writing a line of code
  • Keep your project folder organized — Claude Code can read any file in your project, so clean structure means cleaner context

Context window limits: There's also a ceiling on how much Claude can hold in a single conversation. Very long sessions can cause Claude to "lose track" of things said early on. When a session gets long and results start feeling off, starting a fresh session with a clear re-brief is often faster than continuing.

Writing Prompts That Actually Work

The biggest difference between students who get great results and students who get mediocre results isn't intelligence or technical background — it's prompt quality. Claude is only as good as what you give it to work with.

A good prompt has four parts:

  1. Context — What is this? What have you already built?
  2. Goal — What do you want to happen?
  3. Constraints — Colors, layout, style, what not to change
  4. Reference — Examples, data, or error messages Claude needs to see

Not every prompt needs all four — but the more of them you include, the better the output.

Weak prompts
"Make the chart better"
"Add some interactivity"
"Fix it"
Strong prompts
"The bar chart labels are overlapping when there are more than 6 bars. Rotate the labels 45° and add 20px bottom padding."
"Add a dropdown filter that lets users filter by region (North, South, East, West). Keep the existing color scheme."
"The page is blank. Console error: [error]. The chart worked before I added the filter — I think the issue is in the data filtering function."
The specificity rule: If your prompt could describe a hundred different outcomes, it will produce a random one. If it can only describe one outcome, it will produce that one. Writing a prompt in a notes app first, reading it back, and asking "does Claude have everything it needs?" is worth 30 seconds every time.

Plan Mode — Your Most Valuable Habit

Plan mode is a feature in Claude Code that makes Claude describe what it intends to do before it writes a single line of code. Instead of immediately generating output, it lays out its approach and waits for your approval.

Why this matters: Claude Pro has usage limits. Every message uses some of that limit. If Claude misunderstands a complex request and writes 200 lines of wrong code, you've burned usage on something you have to undo and redo. Plan mode catches misunderstandings before they become expensive mistakes.

To activate it, start your prompt with:

Prompt — Activate Plan Mode
Before you write any code, make a plan. Describe what you're going to build and how, then wait for my approval before starting.

Claude will respond with something like:

"Here's my plan:
1. Add a <select> dropdown above the chart
2. Populate it with the four region options
3. Add an event listener that fires when the selection changes
4. Filter the data array and re-render the chart

Does this look right before I start?"

Now you can catch the misunderstanding before it's written. You can say "actually, also add an 'All Regions' option as the default" and Claude adjusts its plan — instead of you discovering the omission after it's already built.

Use plan mode when:

  • You're adding a significant new feature to an existing dashboard
  • The task involves multiple changes at once
  • You're not 100% sure Claude understands what you're asking
  • You want to make sure it won't break what's already working

Skip plan mode for: simple, single-purpose changes where the request is unambiguous ("change the chart title to 'Q3 Revenue'").

Pro tip: You can also trigger plan mode directly in Claude Code by typing /plan before your prompt. Same effect — Claude shows its plan and waits for a thumbs up before touching your files.

The Iteration Loop

Building with Claude isn't a one-shot operation. It's a loop. The sooner you internalize this, the less frustrated you'll be when the first output isn't quite right — because it almost never is, and that's completely fine.

↺
  1. Describe — Tell Claude what you want. Be specific (see above).
  2. Plan — For complex tasks, ask Claude to plan before building. Catch misunderstandings early.
  3. Build — Claude writes the code. Open the file with Live Server in your browser.
  4. Review — Does it look right? Does it work? What's off?
  5. Refine — Describe exactly what needs to change. Be specific. Go back to step 1.
The most common mistake: Trying to describe everything in one giant prompt. Don't. Build the structure first, then add data, then style, then interactivity. Each step is its own loop. A dashboard built in five focused iterations will be better than one built with one sprawling prompt.

Protecting Your Usage

Claude Pro has usage limits that reset on a rolling basis. Most students have more than enough for this course if they're deliberate. Here's how to use your limit wisely:

  • Think before you send. Write your prompt in a notes app first, read it back, and ask yourself: does Claude have everything it needs? A 30-second pause before sending is worth it.
  • Use plan mode for complex tasks. A planning exchange costs far less than rebuilding something that went wrong.
  • Batch related changes. Instead of three separate messages ("make the title bigger" → "now bold" → "now purple"), send one: "Make the title larger, bold, and color #8b5cf6."
  • Start fresh for new projects. Long, wandering conversations produce worse results and use more of your limit. A clean session with a focused brief is faster.
  • Commit to GitHub when something works. Before making more changes, commit what's working. If something breaks, you can always roll back instead of asking Claude to undo it.

Assignment 2

Before you build dashboards, you need to know what makes a visualization work — and what makes it fail. This module covers the principles that separate data storytelling from data dumping. You'll build several charts along the way.

Why This Comes Before Dashboarding

It's easy to build a dashboard that's full of charts and completely useless. The charts are technically correct but answer no questions, guide no decisions, and waste the reader's time. The principles in this module are the difference between a dashboard that gets used and one that gets ignored.

These aren't design rules — they're communication rules. Visualization is a language. This module teaches you the grammar.

1 — Choose the Right Chart for the Story

The most common visualization mistake is choosing a chart type because it looks interesting, not because it fits the data. Here's the map:

  • Bar chart — comparing values across categories. Best for: "Which region had the highest revenue?" Always use horizontal bars when category labels are long.
  • Line chart — showing change over time. Best for: "How has revenue trended over the past 12 months?" Never use for unordered categories.
  • Scatter plot — showing the relationship between two numeric variables. Best for: "Is there a correlation between marketing spend and sales?"
  • Pie / Doughnut — showing composition (parts of a whole). Use sparingly — only when you have 4 or fewer slices and the proportions are meaningfully different. When in doubt, use a bar chart instead.
  • Area chart — showing cumulative volume over time. Good for: "How much total revenue has accumulated this quarter?"
  • Stacked bar — showing both totals and composition simultaneously. Best for: "How does revenue break down by region, and how do totals compare month to month?"
The pie chart trap: Humans are bad at comparing angles and areas. A bar chart almost always communicates the same information more clearly. Reserve pie charts for when the "part of whole" framing is genuinely the point — e.g., "60% of users are on mobile."
Tutorial 1 — The Right Chart
Build me a single HTML file with three side-by-side charts, all using the same dataset: monthly coffee shop revenue for Jan–Jun (Jan: $12,400, Feb: $10,800, Mar: $14,200, Apr: $15,600, May: $18,900, Jun: $22,300).

Chart 1: A bar chart
Chart 2: A line chart
Chart 3: A pie chart

Use Chart.js via CDN. Dark background. Put a label above each chart with its chart type. Make all three the same size so I can compare them directly.

Don't style them beautifully — I want to see the raw difference in how each type communicates the same data.

After building it: which chart makes the trend most obvious? Which makes it hardest to see? Which one would you never use for this data, and why?

2 — Color with Purpose

Color is one of the most misused tools in data visualization. The default Chart.js rainbow palette looks playful but communicates nothing — each color is just "a different category" with no additional meaning.

There are three meaningful ways to use color:

  • Categorical — distinct colors for distinct groups (regions, teams, product lines). Limit to 5–7 colors. Use colors that are perceptually distinct, not just visually different.
  • Sequential — light → dark of one hue to show quantity (e.g., a heat map of sales density). More color = more value.
  • Diverging — two hues from a neutral midpoint, for data with a meaningful center (e.g., profit vs. loss, above vs. below average).

The accent rule: In most dashboards, use neutral colors for most data and one bright accent color to highlight the most important value. The eye goes to contrast — use that deliberately.

Colorblind accessibility: ~8% of men have red-green color blindness. Avoid using red and green to encode opposing meanings (good vs. bad, up vs. down). Use blue/orange or add icons/patterns as a secondary signal.
Tutorial 2 — Color with Purpose

Then, in the same HTML file, build a second version directly below it using a purposeful color palette: muted blue-gray for Hardware and Services, and a bright accent color (your choice) for Software — because Software is our fastest-growing line and deserves emphasis.

Dark background. Add a short caption below each chart: "Default palette" and "Purposeful palette." I want to see the difference side by side.

3 — The Data-Ink Ratio

Edward Tufte's most useful principle: every pixel on a chart should be doing work. The ratio of "ink that represents data" to "total ink used" should be as high as possible.

Things that don't represent data (remove them):

  • Heavy gridlines — use very faint ones or none at all
  • Chart borders and box outlines
  • 3D effects (never — they distort perception of values)
  • Redundant legends when the data is already labeled directly
  • Decorative background images or gradients behind charts
  • Excessive tick marks on axes

Things that do represent data (keep them):

  • The bars, lines, or points themselves
  • Axis labels (with units)
  • A descriptive title
  • Data labels when precision matters
  • A legend when the colors encode meaningful categories
Tutorial 3 — Data-Ink Ratio
Build me a line chart showing monthly active users for a SaaS product over 12 months: Jan: 4200, Feb: 4800, Mar: 5100, Apr: 5600, May: 6200, Jun: 6800, Jul: 7100, Aug: 7400, Sep: 7200, Oct: 7800, Nov: 8400, Dec: 9100.

Build two versions side by side in the same HTML file:

Version 1 — "Cluttered": Add a visible chart border, heavy gridlines (every 500 units), a background gradient behind the chart area, bold axis tick marks, a large legend box, and a 3D-style drop shadow on the line.

Version 2 — "Clean": Remove the border, use faint gridlines only at major values, remove the legend (label the line directly instead), keep axis labels minimal, no decorative elements. Let the upward trend be the only thing you notice.

Same data, same size. I want to see Tufte's principle in action.

4 — Titles That Make an Argument

A title is not a label. "Monthly Revenue" describes what the chart shows. "Revenue Has Grown 58% Since January — And It's Accelerating" tells the reader what to think about it.

The best data visualization titles are declarative — they make a claim and use the chart as evidence. This is especially important in narrative reports (Module 05), but it applies everywhere.

The formula:

  • Weak: "Sales by Region" — describes the axes
  • Better: "North Region Leads in Sales" — identifies the key finding
  • Best: "North Region Outperforms East and West by 30% — and the Gap Is Widening" — makes an argument with specificity

Always include: units on axes ("Revenue ($K)" not just "Revenue"), the date range in the title or subtitle, and a data source note at the bottom.

Tutorial 4 — Declarative Titles

Then, directly below it, build the exact same chart with a declarative title that makes an argument about what the data shows — something specific that tells the reader what the key insight is. Use a dark theme. Highlight the most notable bar with an accent color.

5 — Lie Factor (Tufte)

Tufte's Lie Factor measures how honest a visualization is:

Lie Factor = size of effect shown in graphic ÷ size of effect in data

A Lie Factor of 1.0 is perfectly honest. Greater than 1.0 means the chart exaggerates the difference. The most common way this happens is a truncated y-axis — starting your bar chart at something other than zero so that small differences look enormous.

For example: if revenue grew from $980k to $1,000k (a 2% increase), but the y-axis runs from $975k to $1,005k, the bar appears to be 5× taller — a Lie Factor of roughly 5. The chart isn't technically wrong, but it's visually deceptive.

This happens constantly in media and corporate presentations. When you see a bar chart in the wild, always check where the y-axis starts. If it doesn't start at zero, ask yourself why — and what impression it creates.
Tutorial 5A — Build a Misleading Chart
Build an HTML file with two side-by-side bar charts showing the exact same data: quarterly revenue for four competitors (Company A: $31.2M, Company B: $30.8M, Company C: $31.0M, Company D: $30.6M).

Chart 1 — "The Honest Version": Y-axis starts at $0. Bars look nearly identical in height.

Chart 2 — "The Deceptive Version": Y-axis starts at $30.4M and ends at $31.4M. Use the same data — only the axis changes.

6 — Small Multiples (Tufte)

Small multiples are one of the most underused patterns in data visualization. The idea: instead of cramming multiple lines or series into a single chart (which creates visual noise), repeat the same chart structure in a grid — one panel per category.

The reader's eye learns the structure once and then scans across panels to compare patterns. It's dramatically easier to read than a chart with 6 overlapping lines.

Rule of thumb: If you have more than 3–4 series on a single chart and they're all important, small multiples is probably the right call.
Tutorial 6A — Spaghetti vs. Small Multiples

Quick Reference — What to Avoid

  • Truncated Y-axes — starting your Y-axis at something other than zero exaggerates differences. A 2% change looks like a 50% change. Always start at zero for bar charts.
  • Dual Y-axes — two Y-axes on one chart makes it easy to suggest a relationship that may not exist by adjusting the scales. Use separate charts instead.
  • 3D charts — always. 3D distorts the visual perception of values. A bar that's "behind" another looks smaller even if it isn't.
  • Pie charts with many slices — anything more than 4-5 slices becomes impossible to read. Group small values into "Other."
  • Correlation as causation — two lines moving together doesn't mean one causes the other. Be careful with your titles and annotations.
  • Too many chart types on one dashboard — using 6 different chart types feels creative but is cognitively exhausting. Limit yourself to 2-3 per dashboard.

What Is an HTML File?

An HTML file is a single text file that any browser can render as a webpage. When we build dashboards, everything lives in one file: the structure (HTML), the styling (CSS), and the interactivity (JavaScript). You can open it locally by double-clicking it, or publish it to GitHub Pages and share the URL with anyone.

You don't write this file. Claude does. Your job is to describe what you want clearly enough that Claude can build it — and then to refine it until it's right.

What Is Chart.js?

Chart.js is a free, open-source JavaScript library that creates beautiful, interactive charts. We use it by loading it from a CDN (content delivery network) — one line of code at the top of our HTML file, no installation required.

Claude knows Chart.js well. When you ask for a bar chart, line chart, doughnut, scatter plot, or radar chart, Claude will use Chart.js to build it.

Working with Data — Two Approaches

Approach 1: Embed data in the HTML — Best for small, static datasets. Claude writes the data directly into the JavaScript as a variable. Simple, portable, works everywhere.

Approach 2: Fetch from GitHub — Best for larger datasets or data you want to update easily. You upload the CSV to your GitHub repo, then the HTML fetches it via the raw URL. This is what we'll do in the projects.

Why not just open a CSV locally? Browsers block local file access for security reasons. By hosting your data on GitHub and fetching it via URL, you bypass this entirely — and get version control for free.

What Is the /dash Skill?

A .md file saved as .claude/commands/dash.md in your GitHub repo. It has two parts: your personal design spec (colors, fonts, layout, chart style) and a planning protocol — instructions that tell Claude to ask you the right questions before writing any code. Claude Code automatically turns any file in .claude/commands/ into a slash command, so dash.md becomes /dash.

Type /dash at the start of a build session and Claude loads your spec, then walks you through planning: what's the goal, who's the audience, what does the data look like? Only after you answer does it produce a build brief and wait for your go-ahead. No copy-pasting. No starting cold. No building the wrong thing.
